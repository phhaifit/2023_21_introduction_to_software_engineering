## Context

The Task & Orchestration module requires comprehensive, secure presentation of optional execution activity (tools, sub-agents, workflows) generated by external OpenClaw runtimes. To ensure strict modularity and security, this observability data must be presented through provider-neutral adapter boundaries, guaranteeing the application domain projects activity without assuming administration of tools, agents, or provider internals. The overarching architectural vision mandates a provider-neutral boundary:
```text
Frontend
    ↓
Task & Orchestration application boundary
    ↓
TaskExecutionAdapter
    ↓
Externally supplied execution runtime reference
    ↓
OpenClaw instance managed by another module/team
```
Workspace Management or the responsible infrastructure module provides a resolvable execution-runtime reference. Task & Orchestration consumes that reference to submit and monitor work.

**Task & Orchestration owns:** Task and TaskWork domain models, Task creation/validation, canonical Task lifecycle (Pending, In-Progress, Completed, Failed, Canceled), routing request representation (Auto, Specific Agent, Predefined Workflow), conversation and Task history, mock execution used for development and tests, provider-neutral execution contracts, consumer-side OpenClaw adapter, mapping platform Task requests to verified OpenClaw requests, mapping OpenClaw execution updates to normalized Task events, cancellation request forwarding, execution reference association, lifecycle projection, streaming/result/error/observability presentation, Task-scoped event isolation, frontend rendering and user interaction.

**Task & Orchestration does not own:** OpenClaw installation, OpenClaw container creation, OpenClaw start/stop/restart/delete/upgrade, workspace provisioning, CPU/RAM allocation, Standard/Premium infrastructure configuration, Gateway DNS or networking, Gateway credential creation or platform-wide secret ownership, authentication implementation, workspace membership management, RBAC ownership, subscription validation implementation, payment, Agent Management, Workflow Management, Tool Management, Knowledge Base or RAG Management, custom orchestration engine, custom LLM router, custom multi-agent collaboration engine, custom workflow runtime, OpenClaw internals.

## Goals / Non-Goals

**Goals:**
- Present safe, normalized execution activity received from OpenClaw through the Task & Orchestration integration boundary.
- Support optional presentation of routing activity, workflow activity, tool activity, sub-agent activity, handoff, review, aggregation, completion, and provider diagnostics.
- Ensure Task & Orchestration only displays or projects activity supplied by the provider without creating tools, assigning tools to agents, creating sub-agents, controlling OpenClaw internal orchestration, creating workflows, inferring unprovided events, or exposing raw credentials/sensitive payloads.
- Enforce graceful degradation: `If optional provider observability events are unavailable, the canonical Task lifecycle SHALL remain functional.`
- Implement event-scoping requirements using workspace ID, Task ID, Work ID, execution reference, and provider session/run reference when available.
- Enforce safe redaction requirements and ensure Advanced Details show provider references only when authorized and safe.
- Reframe mock execution as a legitimate test and development adapter.

**Non-Goals:**
- Tool creation, Tool Management, Agent Management, or Workflow Management administration.
- OpenClaw internal orchestration control or custom collaboration engines.
- Runtime provisioning, container creation, secret ownership, or direct OpenClaw API invocations.

## Decisions

### Decision 1: Observability as Projection Only
Task & Orchestration acts strictly as an observability projection consumer. It displays activity supplied by the provider but never creates tools, assigns tools to agents, creates sub-agents, controls OpenClaw internal orchestration, creates workflows, or infers events that were not provided.
* *Rationale*: Maintains absolute decoupling between platform task monitoring and provider execution intelligence.
* *Alternatives considered*: Task & Orchestration dynamically registering new tools inside OpenClaw during task start. Rejected because it violates Tool Management ownership.

### Decision 2: Graceful Degradation of Observability
If optional provider observability events are unavailable, the canonical Task lifecycle SHALL remain functional. The absence of detailed tool or sub-agent diagnostics must never halt task progression or be interpreted as a task failure.
* *Rationale*: Ensures robust enterprise execution where lightweight runtimes or restricted environments may omit verbose diagnostic streams.
* *Alternatives considered*: Failing the task if the provider fails to emit tool activity milestones. Rejected because it introduces unnecessary fragility into the core lifecycle.

### Decision 3: Event-Scoping Isolation Boundaries
Observability events are strictly scoped using workspace ID, Task ID, Work ID, execution reference, and provider session/run reference when available.
* *Rationale*: Prevents cross-task data leakage and guarantees multi-tenant security in shared conversation workspaces.
* *Alternatives considered*: Broadcasting un-scoped provider logs globally to the frontend. Rejected because it introduces catastrophic data leakage between workspaces.

### Decision 4: Automated Security Redaction
The adapter implements automated security redaction filters, scrubbing raw credentials, API keys, system paths, and sensitive provider payloads before events reach the presentation layer. Advanced Details show provider references only when authorized and safe.
* *Rationale*: Prevents sensitive system secrets and infrastructure configurations from leaking into client-side DOM trees or processing inspectors.
* *Alternatives considered*: Relying on the frontend to hide sensitive JSON keys via CSS. Rejected because the data still exists in the browser memory and network tabs.

### Decision 5: External Metadata Dependencies
Tool, agent, and workflow metadata may be supplied by their owning modules. Task & Orchestration may display safe labels but must not become the source of truth for their administration.
* *Rationale*: Enforces clear modularity and single sources of truth across the platform.
* *Alternatives considered*: Task & Orchestration maintaining its own database tables for Tool definitions. Rejected because it duplicates Tool Management capabilities.

### Decision 6: External Dependency Contracts & Responsibility Boundaries
The Task & Orchestration specifications define consumer-facing ports conceptually but must not claim ownership of their implementations:
```ts
interface WorkspaceExecutionRuntimeResolver {
  resolve(workspaceId: WorkspaceId): Promise<WorkspaceExecutionRuntime>;
}

interface WorkspaceExecutionRuntime {
  provider: "openclaw";
  instanceId: string;
  endpointReference: string;
  credentialReference: string;
  status: "running" | "stopped" | "unavailable";
}
```
External dependencies are defined conceptually for:
- Agent Management → workspace-scoped selectable agents, platform-agent to provider-agent mapping
- Workflow Management → workspace-scoped selectable workflows, platform-workflow to provider-workflow mapping
- Workspace User Management / Authentication → authorized principal and operation permission
- Workspace Management → resolvable execution-runtime reference

The cross-change dependency order is documented as follows:
```text
enhance-task-orchestration-production-ui
    independent presentation alignment

establish-openclaw-task-integration-contracts
    defines consumer-side contracts

integrate-openclaw-task-execution
    depends on integration contracts
    and on an externally supplied runtime prerequisite

extend-openclaw-execution-observability
    depends on task execution integration
```
* *Rationale*: Prevents scope creep and enforces strict architectural boundaries across project modules.
* *Alternatives considered*: Implementing concrete container health checks inside Task & Orchestration. Rejected because it violates module ownership.

## Risks / Trade-offs

* **Risk: Verbose sub-agent activity payloads overwhelm frontend rendering** → Mitigation: Advanced Details are closed by default, and the timeline aggregates sub-agent steps cleanly without re-rendering the entire feed.
* **Risk: External module metadata catalogs are temporarily unreachable** → Mitigation: Task & Orchestration gracefully falls back to displaying raw, safe target IDs without halting the canonical task lifecycle.

## Migration Plan

1. Extend `NormalizedRuntimeEvent` union contracts to include optional observability payloads.
2. Implement automated security redaction filters and event-scoping isolation logic in the adapter.
3. Establish conceptual consumer ports for Tool, Agent, and Workflow metadata catalogs.
4. Verify strict validation of all OpenSpec artifacts and confirm no provisioning logic exists in Task & Orchestration.
