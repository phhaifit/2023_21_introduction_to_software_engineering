## Why

### Problem
The Task & Orchestration module requires comprehensive, secure presentation of optional execution activity (tools, sub-agents, workflows) generated by external OpenClaw runtimes. To ensure strict modularity and security, this observability data must be presented through provider-neutral adapter boundaries, guaranteeing the application domain projects activity without assuming administration of tools, agents, or provider internals.

### Goals
This change must:
- Present safe, normalized execution activity received from OpenClaw through the Task & Orchestration integration boundary.
- Support optional presentation of routing activity, workflow activity, tool activity, sub-agent activity, handoff, review, aggregation, completion, and provider diagnostics.
- Ensure Task & Orchestration only displays or projects activity supplied by the provider without creating tools, assigning tools to agents, creating sub-agents, controlling OpenClaw internal orchestration, creating workflows, inferring unprovided events, or exposing raw credentials/sensitive payloads.
- Enforce graceful degradation: `If optional provider observability events are unavailable, the canonical Task lifecycle SHALL remain functional.`
- Implement event-scoping requirements using workspace ID, Task ID, Work ID, execution reference, and provider session/run reference when available.
- Enforce safe redaction requirements and ensure Advanced Details show provider references only when authorized and safe.
- Reframe mock execution as a legitimate test and development adapter.

## What Changes

### Scope of Changes
- Extend `NormalizedRuntimeEvent` union to support optional observability payloads (routing activity, workflow activity, tool activity, sub-agent activity, handoff, review, aggregation, completion, provider diagnostics).
- Enforce strict projection-only boundary ensuring Task & Orchestration never creates tools, assigns tools, creates sub-agents, controls OpenClaw internal orchestration, or infers events that were not provided.
- Establish graceful-degradation requirement ensuring canonical Task lifecycle remains 100% functional when optional observability events are missing.
- Define event-scoping isolation boundaries utilizing workspace ID, Task ID, Work ID, execution reference, and provider session/run reference.
- Implement automated security redaction filters scrubbing raw credentials, API keys, and sensitive provider payloads before presentation.
- Establish permission-gated rendering for Advanced Details ensuring provider diagnostics are shown only when authorized and safe.

### Responsibility Boundary & Preconditions
The specifications consistently reflect the following architecture:
```text
Frontend
    ↓
Task & Orchestration application boundary
    ↓
TaskExecutionAdapter
    ↓
Externally supplied execution runtime reference
    ↓
OpenClaw instance managed by another module/team
```
Workspace Management or the responsible infrastructure module provides a resolvable execution-runtime reference. Task & Orchestration consumes that reference to submit and monitor work.

**External Dependencies:** Tool, agent, and workflow metadata may be supplied by their owning modules. Task & Orchestration may display safe labels but must not become the source of truth for their administration.

**Task & Orchestration owns:** Task and TaskWork domain models, Task creation/validation, canonical Task lifecycle (Pending, In-Progress, Completed, Failed, Canceled), routing request representation (Auto, Specific Agent, Predefined Workflow), conversation/Task history, mock execution used for development/tests, provider-neutral execution contracts, consumer-side OpenClaw adapter, mapping platform Task requests to verified OpenClaw requests, mapping OpenClaw execution updates to normalized Task events, cancellation request forwarding, execution reference association, lifecycle projection, streaming/result/error/observability presentation, Task-scoped event isolation, frontend rendering and user interaction.

**Task & Orchestration does not own:** OpenClaw installation, OpenClaw container creation, OpenClaw start/stop/restart/delete/upgrade, workspace provisioning, CPU/RAM allocation, Standard/Premium infrastructure configuration, Gateway DNS or networking, Gateway credential creation or platform-wide secret ownership, authentication implementation, workspace membership management, RBAC ownership, subscription validation implementation, payment, Agent Management, Workflow Management, Tool Management, Knowledge Base or RAG Management, custom orchestration engine, custom LLM router, custom multi-agent collaboration engine, custom workflow runtime, OpenClaw internals.

### Explicit Out of Scope
This change explicitly does not implement:
- Tool creation, Tool Management, Agent Management, or Workflow Management administration;
- OpenClaw internal orchestration control or custom collaboration engines;
- Runtime provisioning, container creation, or secret ownership;
- Direct OpenClaw API invocations bypassing adapter boundaries.

## Capabilities

### New Capabilities
- `openclaw-execution-observability`: Establishes the optional observability projection contracts, safe activity rendering rules, graceful degradation mechanisms, event-scoping isolation boundaries, and automated security redaction filters without provisioning runtimes or administering external module domains.

### Modified Capabilities
- `task-execution-adapter`: Extends the adapter DTO definitions and event union mapping to accommodate optional tool, workflow, and sub-agent diagnostic payloads securely.

## Impact

### Architectural Impact
- **Observability Presentation**: Refines the processing inspector and timeline components in `apps/frontend/src/features/task-orchestration/components` to project safe, normalized activity without storing credentials or inferring unprovided events.
- **Domain Decoupling**: Ensures Tool, Agent, and Workflow metadata are consumed from external catalogs, preserving strict module ownership.

### External Dependencies and Cross-Change Boundaries
The Task & Orchestration module defines consumer-facing ports conceptually but does not claim ownership of their implementations:
```ts
interface WorkspaceExecutionRuntimeResolver {
  resolve(workspaceId: WorkspaceId): Promise<WorkspaceExecutionRuntime>;
}

interface WorkspaceExecutionRuntime {
  provider: "openclaw";
  instanceId: string;
  endpointReference: string;
  credentialReference: string;
  status: "running" | "stopped" | "unavailable";
}
```
External dependencies are defined conceptually for:
- Agent Management → workspace-scoped selectable agents, platform-agent to provider-agent mapping
- Workflow Management → workspace-scoped selectable workflows, platform-workflow to provider-workflow mapping
- Workspace User Management / Authentication → authorized principal and operation permission
- Workspace Management → resolvable execution-runtime reference

The cross-change dependency order is documented as follows:
```text
enhance-task-orchestration-production-ui
    independent presentation alignment

establish-openclaw-task-integration-contracts
    defines consumer-side contracts

integrate-openclaw-task-execution
    depends on integration contracts
    and on an externally supplied runtime prerequisite

extend-openclaw-execution-observability
    depends on task execution integration
```
