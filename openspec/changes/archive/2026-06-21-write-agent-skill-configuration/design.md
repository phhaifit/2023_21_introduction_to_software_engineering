## Context

Agent Management has completed Phase 1–4 (App Shell, HTTP API, UI↔API, Prisma Persistence). The current architecture is hexagonal:

- **Port:** `AgentRepository` interface — persistence adapter
- **Application:** `AgentLifecycleUseCases` — orchestrates create/update/enable/disable/delete
- **Existing skill config:** `generateAgentSkillConfiguration(agent)` — pure function that returns a markdown string
- **Current gap:** The generated string is returned in `AgentMutationResult` but never written to disk. The API router discards it.

## Goals / Non-Goals

**Goals:**
- Define `AgentSkillWriter` as a new port interface in the application layer.
- Implement `FileSystemAgentSkillWriter` that writes `skill.md` files using `node:fs/promises`.
- Implement `NoOpAgentSkillWriter` for tests.
- Inject the writer into `AgentLifecycleUseCases` and call it from `saveMutation`.
- Skill write failures are logged but do not block the mutation (fire-and-forget with error logging).
- Directory structure: `<AGENT_SKILLS_DIR>/<workspaceId>/<agentId>/skill.md`.

**Non-Goals:**
- Do not call Docker, OpenClaw, or any external runtime directly.
- Do not change the `Agent` domain model.
- Do not change HTTP API contracts.
- Do not handle skill file deletion when an agent is deleted (future phase).

## Decisions

### 1. Port Interface — `AgentSkillWriter`

File: `backend/src/modules/agent-management/application/agent-skill-writer.ts`

```typescript
import type { Agent } from "../domain/agent.ts";

export type AgentSkillWriter = {
  writeSkillConfiguration(agent: Agent, content: string): Promise<void>;
};
```

**Rationale:** Follows the same pattern as `AgentRepository` — a plain TypeScript type, not a class. Receives both the agent (for path resolution) and the content string (already generated by `generateAgentSkillConfiguration`).

### 2. File System Implementation

File: `backend/src/modules/agent-management/infrastructure/file-system-agent-skill-writer.ts`

```typescript
import { mkdir, writeFile } from "node:fs/promises";
import { join } from "node:path";
import type { Agent } from "../domain/agent.ts";
import type { AgentSkillWriter } from "../application/agent-skill-writer.ts";

export class FileSystemAgentSkillWriter implements AgentSkillWriter {
  constructor(private readonly baseDir: string) {}

  async writeSkillConfiguration(agent: Agent, content: string): Promise<void> {
    const dir = join(this.baseDir, agent.workspaceId, agent.agentId);
    await mkdir(dir, { recursive: true });
    await writeFile(join(dir, "skill.md"), content, "utf-8");
  }
}
```

**Rationale:**
- `mkdir({ recursive: true })` creates parent directories if they don't exist.
- Path structure `<baseDir>/<workspaceId>/<agentId>/skill.md` ensures workspace isolation.
- Uses `node:fs/promises` — no external dependencies needed.

### 3. No-Op Implementation

File: `backend/src/modules/agent-management/infrastructure/no-op-agent-skill-writer.ts`

```typescript
import type { AgentSkillWriter } from "../application/agent-skill-writer.ts";

export class NoOpAgentSkillWriter implements AgentSkillWriter {
  async writeSkillConfiguration(): Promise<void> {}
}
```

### 4. Use Cases Integration

File: `backend/src/modules/agent-management/application/agent-lifecycle-use-cases.ts`

Changes:
- Add `skillWriter?: AgentSkillWriter` to `AgentLifecycleDependencies`.
- In `saveMutation`, after `repository.save`, call `this.dependencies.skillWriter?.writeSkillConfiguration(saved, skillConfiguration)` wrapped in try/catch.
- Log errors via `console.error` if the writer fails — do not rethrow.

```typescript
private async saveMutation(agent: Agent): Promise<AgentMutationResult> {
  const saved = await this.dependencies.repository.save(agent);
  const skillConfiguration = generateAgentSkillConfiguration(saved);

  try {
    await this.dependencies.skillWriter?.writeSkillConfiguration(saved, skillConfiguration);
  } catch (err) {
    console.error(`Failed to write skill configuration for agent ${saved.agentId}:`, err);
  }

  return {
    agent: saved,
    publicSummary: toAgentPublicSummary(saved),
    skillConfiguration
  };
}
```

**Rationale:**
- Optional chaining (`?.`) means existing tests without a writer still work without changes.
- Fire-and-forget with error logging ensures that a filesystem failure doesn't break agent creation.
- Existing `AgentMutationResult` type and `generateAgentSkillConfiguration` are reused unchanged.

### 5. Composition Root Update

File: `backend/src/local-agent-management-server.ts`

- Check `process.env.AGENT_SKILLS_DIR`.
- If set, create `FileSystemAgentSkillWriter` with that directory.
- If not set, use `NoOpAgentSkillWriter` (or omit the dependency — both work with optional chaining).
- Pass `skillWriter` to `AgentLifecycleUseCases`.

### 6. Test Strategy

- **Existing tests:** No changes needed — `skillWriter` is optional, existing lifecycle tests continue to pass.
- **Unit test (writer invocation):** New test `tests/contract/agent-management-skill-writer.test.mjs`:
  - Mock writer tracks calls.
  - `createAgent` → assert writer called once with correct agent and content.
  - `updateAgent` → assert writer called once with updated content.
  - Writer throws → assert agent is still saved, no error propagates.
- **Integration test (file system):** New test `tests/contract/agent-management-skill-file-system.test.mjs`:
  - Uses a real temp directory (`os.tmpdir`).
  - Write → read back → assert content matches.
  - Overwrite → assert content updated.
  - Missing directory → assert created automatically.
  - Cleanup temp files after test.

## Risks / Trade-offs

- **Risk:** File system permissions on the target directory.
  **Mitigation:** The writer uses `mkdir({ recursive: true })` which handles most cases. CI/CD environments can use `NoOpAgentSkillWriter`.
- **Trade-off:** Fire-and-forget means a failed write could go unnoticed.
  **Rationale:** For Phase 5, logging is sufficient. A retry/monitoring system can be added in a future phase.
- **Trade-off:** No skill file cleanup on agent deletion.
  **Rationale:** Out of scope for this phase — will be addressed when deletion semantics are refined.
